00 / 背景

「並列で走らせれば速い」は、たいてい本当。
ただし、いくつかの落とし穴を踏まなければ。

Claude Code を1台動かしているうちは平和でした。 ところが2台、3台と同じリポジトリで動かしはじめた瞬間、 ファイルが取り合いになり、書いたはずのコードが消え、コンフリクトの赤い文字に時間を溶かす ── そんな話が社内で増えてきました。この資料は、その「最初の地雷原」を一緒に歩くための地図です。

「Gitのコマンドを覚える必要はありません。
ただ、何が起きているかと、何を絶対にしないかだけは、
肌感としてわかっていてほしい。」

この資料が答えるもの

  • 並列でClaudeを走らせると、なぜ事故が起きるのか
  • 「ブランチを分ける」が、なぜ最初の正解なのか
  • 初心者がやりがちな6つの危険操作
  • PRとマージは、結局なにを依頼しているのか
  • 人間が介在しなくても安全に走り続ける「ハーネス」とは
01 / 並列開発の正体

同じ家に、3人の
フリーランスを同時に呼ぶこと。

Claude Code を並列で動かす、というのは、ざっくり言えば同じプロジェクトに3人の腕利きフリーランスを同時に呼んで、それぞれ違う仕事を任せるような状況です。 速い。でも、段取りを決めずに呼ぶと、3人がリビングのテーブルを取り合います。

A — 1台運用

一人で全部やる

順番に頼めば衝突しない。安全だけど、4時間かかる仕事は4時間かかる。

B — 並列・無策

同じテーブルで3人が書く

同じ書類を3人が同時に編集。後から書いた人が前の人の変更を上書きしてしまう。

C — 並列・分業

机を3つに分ける

それぞれの机で別々の章を担当。完成したら順番に1冊に綴じる。これが理想形。

02 / コンフリクトとは何か

同じ行を、2人が違うふうに書き直した。
そのときGitは黙ってどちらかを選びません。

コンフリクトは、Google ドキュメントで同じ段落を二人がほぼ同時に編集したときの状況に似ています。 ただし Git はオフラインで使えるぶん、その場で衝突を伝えてくれません。 「合流するタイミング（=マージ）」になって初めて、 「ここ、二人が違う事を書いてるけど、どっち採用？」と、人間に判断を委ねてきます。

厄介なのは、「コンフリクトが出る」こと自体は事故ではない点です。 並列で開発していれば、ある程度は必ず起きます。ですから身も蓋もない結論を先に書きます ── コンフリクトを完全に防ぐ努力に時間を使うより、出てきたコンフリクトと丁寧に向き合うことに時間を使ってください。

一番危ないのは、「なんかよくわかんないけど、こっちっぽい」で済ませてしまうこと。 これで誰かの仕事が静かに消えます。わからなければ、わかるまで Claude に 「これは何と何が衝突していて、どちらを残すとどうなるのか、私が判断できるように説明して」と頼んでください。 納得してから解消する。これだけで、ほとんどの事故は防げます。

そして並列開発の段取りとしては、そもそも作業フォルダを物理的に分ける worktree という仕組みが、 衝突の発生頻度自体を下げてくれます。次の章で扱います。

03 / 解決策

worktree =「机を物理的に分ける」発想。

並列開発の事故の99%は、「同じ作業フォルダで複数のClaudeを動かす」ことから来ます。 解決は驚くほど単純で、Claudeごとに作業フォルダを分けてしまう。 これを Git の世界では worktree（ワークツリー） と呼びます。 設計図で言えば「同じ建物のフロアごとに、別々の現場詰所を持つ」イメージです。

並列の本物は、ターミナルを並べる × 机を分ける、の合わせ技。

ここで一つ誤解を解いておきます。 1つのターミナルで Claude を動かしたまま「並列でやって」と頼んでも、それは並列にはなりません。 Claude は1つのターミナルの中では、結局ブランチを切り替えながら順番に作業するだけだからです。

本物の並列にする条件は2つ。ターミナルを複数開くこと、そして各ターミナルで worktree（作業フォルダ）を別々に持つこと。 この2つが揃って初めて、3つの机が物理的に分かれた状態になります。

Desk 01
ログイン機能の改修

Claudeが担当。認証まわりのファイルだけを触る。他の机からは見えない。

branch · feature/auth
Desk 02
検索ページの新設

別のClaudeが担当。検索のページだけを書く。ログイン側には一切影響なし。

branch · feature/search
Desk 03
ドキュメント整理

3台目のClaudeが担当。README をリライト。他の機能とは完全に独立。

branch · chore/docs
人間がやること / Claudeがやること
ターミナル①  →  worktreeフォルダA  →  Claude① が feature/auth を担当
ターミナル②  →  worktreeフォルダB  →  Claude② が feature/search を担当
ターミナル③  →  worktreeフォルダC  →  Claude③ が chore/docs   を担当

↑ 「ターミナルを開いて並べる」だけが人間の仕事。 worktree を切る、ブランチを作る、といった Git のコマンド自体は Claude が裏でやってくれます。

覚えるルールは一つ。「1ターミナル ＝ 1机（worktree）＝ 1Claude ＝ 1ブランチ」を1対1で揃えること。 この比率が崩れた瞬間 ── 同じフォルダで2つのClaudeを動かす、同じブランチを2つの机で開く ── にコンフリクトや事故が一気に増えます。

もう一つだけコツを挙げるなら、担当を「ファイル単位」で重ねないこと。 同じ package.json のような共有ファイルを、3つの机が同時に書こうとすれば、結局合流時にぶつかります。 Claude には「触るフォルダの範囲」を最初に指定するのが事故率を一気に下げるコツです。

04 / 基本フロー

毎回これの繰り返し。
たった4ステップ。

Git のコマンドは覚えなくていい、というのが本資料の前提です。 下の4ステップを「Claudeに自然な日本語で指示する」だけ。 各ステップに「Claudeに言うなら」の例文を添えました。コピペでそのまま使えます。

STEP 01

机を分ける
— BRANCH

これから何の仕事をするか名前を付け、その専用ブランチを切ってもらう。机を一つ用意するイメージ。

「feature/login-fix という名前で
ブランチ作って、そこに移動して」
STEP 02

仕事を進める
— COMMIT

区切りのいいところで「セーブ」をする。ゲームのセーブポイントと同じで、いつでもこの時点に戻れる。

「ここまでの変更を、
意味のわかるメッセージで
こまめにコミットして」
STEP 03

提案する
— PULL REQUEST

「この変更、本体に取り込んでいい？」とお伺いを立てる。レビュー用の校正依頼みたいなもの。

「今のブランチで、
PRをGitHubに作って。
タイトルと要約も書いて」
STEP 04

合流させる
— MERGE

PRが上がったらレビュー専門のAgentに読ませて、クリティカルな指摘を潰す。問題がなければマージ。人間が1行ずつ追うのではなく、Agentに何をどこまで見させるかを設計する側に時間を使う。

「このPR、レビュー専門の
Agentに渡してチェックさせて。
クリティカルな指摘がなければ
mainにマージして、
ブランチも削除して」
05 / 注意ポイント

並列開発で、
特に気をつけたい6つのこと。

どれも事故に直結しがちな項目ですが、知っていれば避けられるものばかりです。 Claudeに作業を頼むときも、「ここは慎重にお願い」と一言添えておくとリスクが大きく下がります。

01

main ブランチで直接作業しないように気をつける

本番に直結する原本に、いきなり下書きを上書きしてしまう状態。最悪、一発で本番が壊れることがあります。

最初にブランチを切る習慣をつける。Claudeにも「mainで直接編集しないで」と最初に伝えておくとほぼ防げる。
02

force push（強制的に上書き保存）は特に慎重に

たとえると、他の人が共有ドキュメントに書き込んだ履歴ごと、自分のファイルで上書きする操作。チームの作業が静かに消えることがあります。

本当に必要か必ず一度立ち止まる。Claudeには「force pushは慎重に」とCLAUDE.mdに書いておく。
03

同じファイルを複数Claudeが同時に触らないようにする

worktreeで机を分けても、同じファイルを2台が触れば結局衝突します。設定ファイルや型定義は特に注意したい場所。

最初に「あなたが触るフォルダはここまで」と範囲を明確に伝える。共有ファイルは担当を1台に絞っておく。
04

ブランチを長く放置しないようにする

1週間以上分岐したままにしておくと、本体が先に進みすぎてマージ時の衝突が指数関数的に増えていきます。

ブランチは「3〜5日以内にマージ」を目安に短命に保つ。長引きそうなら、一度本体を取り込み直す。
05

Claudeの「ちょっと整理しといた」を鵜呑みにしない

頼んでいない関数の削除、importの整理、設定の書き換えが紛れ込むことがあります。レビューをかけずにマージすると気づけません。

PRの差分は、書いたのとは別のサブエージェントや別モデルにレビューさせる。「想定外の変更があれば指摘して」と明示的に頼む。
06

ブランチを消す前に確認するクセをつける

「もうマージしたから」と削除したブランチに、実は他のClaudeの未マージの差分が残っていた、というケース。

削除前に「未マージの差分がないか確認して」と一言。GitHubのUI上で安全に削除するのが基本。
06 / PRの正体

PR は「校正依頼つきの差し替え提案書」です。

Pull Request（プルリクエスト、PR）という言葉に身構えがちですが、 中身は 「この変更、本体に取り込んでも大丈夫ですか？」 という1枚の提案書です。 Google ドキュメントの「提案モード」と本質的に同じ。 ただし読んで判断する側は、人間とは限りません。 JDDLでは、最初に差分を読むのはレビュー専門の別Agent。 人間は、その指摘の重要度を判断する側に回ります。

PULL REQUEST · #142 · OPEN
feature/auth: ログイン失敗時のメッセージを日本語化
claude-01 が main にマージしたい変更 · 6 files changed (+82 −41)

## 目的
ログインに失敗したときのエラーが英語のままで、サポートに「読めない」と問合せが多いため日本語化しました。

## 確認してほしい点
・文言が他画面のトーンと揃っているか
・パスワード再設定への導線が消えていないか

  • 何が変わる？を一目でわかるタイトル 「ログイン失敗時のメッセージを日本語化」のように、結果がイメージできる短い一文。
  • なぜやったか／何を確認してほしいか レビューする人が3分で文脈を掴めるよう、本文に「目的」「確認ポイント」だけは入れる。
  • 差分は、別の頭にレビューさせる PRを書いたClaudeとは別のサブエージェントや別モデルに、差分を渡して読ませる。同じ頭で書いて見直すより、第三者の視点を一度挟む方が見落としが減る。
  • クリティカルな指摘が潰れたらマージ 本番に反映されるのはここから。マージを押す ＝「責任を持って取り込みます」のサイン。レビューAgentが上げた指摘の重要度を判断するのが、人間の出番。
Column / Opinion

Human-in-the-loop を、
意識的に減らす。

— 自律稼働とハーネスの話

生産性が一段上がる瞬間は、たいてい「人間が判断のたびに止めなくてよくなった」時に来ます。 言い換えれば、人間がループに入る回数を減らせば減らすほど、開発は速くなる。 Andrej Karpathy が2026年に公開した「AutoResearch」は、寝ている2日間で700回の実験を勝手にやり遂げました。介在ポイントは「ゴール設定」と「結果の確認」だけ。

人間を外す、ではなく、人間が出てこなくても安全に走れる「枠」を作る。
その枠の良し悪しが、エンジニアリングの腕の見せどころになる。

とはいえ、ハーネス（安全枠組み）なしに放置すると、Claudeは「いい感じに」削除してはいけないファイルを掃除します。 Anthropic の2026年4月の分析でも、AI エージェントの失敗のほとんどは「権限の過剰委譲」──つまり、やっていい範囲を曖昧にしたまま走らせたことが原因と報告されています。

ハーネスを作るとは、Claude が走り回ってもケガをしない柵を組むこと。具体的には、こういう要素の積み重ねです。

  • worktree で物理分離 事故が他に飛び火しない。最悪、机ごと捨てればよい。
  • 触っていい範囲の明示 「このフォルダの外は読むだけ」とCLAUDE.mdで宣言する。
  • テストで自動検証 完了の定義を「テストが緑」に統一。人間の目視を最後にだけ呼ぶ。
  • ブランチ保護とPR必須 main は人間が承認するまで変えられない設定にしておく。
  • 危険操作の事前ブロック force push / 削除系コマンドは確認なしで走らない設定にする。
  • 長時間タスクは出力を残す 後で何をやっていたか追えるようにログを書かせる。

目指したいのは 「人間を減らす」ではなく「人間を、本当に判断が必要な瞬間にだけ呼ぶ」こと。 毎回呼ばれていたら、エージェントを使う意味は半分になります。 その代わり、呼ばれた時に判断できるよう、私たちは Git・PR・テスト・権限の枠組みづくり側に時間を投資する ── これが、JDDLとしての並列開発の流儀にしていきたい考え方です。

08 / 用語集

この資料で出てきた言葉、ぜんぶ。

アルファベットや英語のままでよく出てくる用語を、日常のたとえとセットで並べました。困ったらここに戻ってきてください。

repositoryリポジトリ
プロジェクトのすべてを入れておく箱。コードも、過去の編集履歴も、誰がいつ何を変えたかもまとめて保管されている。 Google Drive のフォルダに「過去のバージョン全部つき」がついてきたもの。
branchブランチ
本筋（main）から分岐させた、別世界の作業用ライン。ここでなら本筋を壊さずに何でも試せる。完成したら本筋に合流させる。 映画の「もしも」のスピンオフ。本編に影響なく、別ストーリーを進められる。
mainmain ブランチ
本筋・原本。通常、本番環境に反映されるのはこのブランチ。直接編集せず、必ずブランチを切ってからPR経由で取り込む。 出版される「正本」。下書きは別の机でやって、最終OK後にだけここに反映する。
commitコミット
「ここまでをセーブ」する操作。意味のある区切りで打つのが基本。各コミットにはメッセージを添える。 ゲームのセーブポイント。あとから「あのセーブの時に戻りたい」ができる。
mergeマージ
別のブランチを本筋に合流させる操作。中で違う編集があった行は「コンフリクト」になり、人間の判断が要る。 2本の小説の原稿を1冊に綴じる作業。同じ章を別々に書いていたら、どちらを採用するか決める必要がある。
conflictコンフリクト
同じ場所を二人が違うふうに書き直していたときに発生する「どっち採用？」の状態。Gitは勝手に決めず、人間に判断を委ねる。 Googleドキュメントで同じ段落を二人がほぼ同時に編集したときに出るアレ。
pull request / PRプルリクエスト
「このブランチの変更を、本筋に取り込んでもいいですか？」という提案書。レビューを受け、OKが出てからマージされる。 校正依頼。「ここをこう直しました、見てもらえますか」を1枚にまとめたもの。
worktreeワークツリー
同じリポジトリを、複数のブランチで「別フォルダとして同時に開く」ための仕組み。Claudeを並列で動かすときの定番テクニック。 同じ建物のフロアごとに、別々の現場詰所を構える。お互いに邪魔し合わない。
push / pullプッシュ／プル
push＝自分の手元の変更をGitHub（クラウド）に上げる。pull＝GitHub上の最新を手元に下ろす。 クラウドストレージのアップロード／ダウンロードと同じ感覚。
force push強制プッシュ
通常のpushが拒否される状況で、上書きを強行する操作。他人の変更を消す可能性があり、初心者は触らないのが鉄則。 「あなたが最後に保存した版で他の人の編集ごと上書きします」というボタン。
remote / localリモート／ローカル
ローカル＝自分のパソコン内。リモート＝GitHub上（チームで共有している方）。push/pullでこの2つを同期させる。 手元の手帳と、クラウド上の共有ノート。書き写しあって整合をとる。
branch protectionブランチ保護
mainなど大事なブランチに「PR経由でしか変更を入れられない」「force pushを禁止」などのルールをかけておく機能。 大事な原稿に「校了印が押されないと差し替え不可」のスタンプを貼っておくこと。
HITLHuman in the loop
自動化された処理の途中で、人間が判断や承認を挟むこと。安全だが、入りすぎると速度が落ちる。バランスが論点。 自動運転の途中で「ここからは手動で運転してください」と言われるアレ。
harnessハーネス
自律的に動くものを「安全に走らせる」ための枠組み。柵、命綱、規約、テスト、権限制限などの総称。 アスレチックでつける命綱。高い場所で自由に動いてもらうために、落ちない仕組みを先に用意する。
